1、Hive表设计优化

1.1 Hive基本查询原理

Hive的设计思想是通过元数据解析描述将HDFS上的文件映射成表

基本的查询原理是当用户通过HQL 语句对Hive中的表进行复杂数据处理和计算时，默认将其转换为分布式计算MR 程序对HDFS中的数据进行读取处理的过程。

减少全表扫描和分区裁剪

当执行查询计划时，Hive会使用表的最后一级目录作为底层的处理数据的输入

Step1: 先根据表名在元数据中进行查询表对应的HDFS目录

当执行查询计划时，Hive会使用表的最后一级目录作为底层的处理数据的输入

Step2: 然后将整个HDFS中表的目录作为底层的查询的输入，可以通过explain命令查看执行计划依赖的数据

1.2 分区表结构-分区设计思想

普通表结构问题

1）假设每天有1G的数据增量，一年就是365GB的数据，按照业务需求，每一次只需要对其中的一天的数据进行处理，也就是处理1GB的数据

2）程序会先加载365GB的数据，然后将364GB的数据过滤掉，只保留一天的数据进行计算，导致了大量的磁盘和网络的IO的损耗

上面的案例中，按照登录日期进行分区存储到Hive表中，每一天一个分区，在HDFS的底层就可以自动的实现将每天的数据存储在不同的目录中

案例：基于分区表的设计实现将所有用户的登录信息进行分区存储

查询先检索元数据，元数据中记录该表为分区表并且查询过滤条件为分区字段，所以找到该分区对应的HDFS目录，通过元数据和执行计划都是可以进行查看的，减少查询时候的全表扫描，分区字段选择也很重要

1.3 分桶表的设计和使用
1.3.1 Hive中Join的问题

默认情况下，Hive底层是通过MR来实现的，MapJoin，ReduceJoin，其中MapJoin效率很高

如果有两张非常大的表进行join，底层无法使用MapJoin来提高Join的性能，只能走默认的ReduceJoin

ReduceJoin必须经过Shuffle过程，相对性能较差，而且容易产生数据倾斜

分桶表的设计思想

分区表是将数据划分不同的目录进行存储，而分桶是将数据划分不同的文件进行存储

分桶表的设计是按照一定的规则，底层通过MR中的Reduce来实现，将数据划分到不同的文件进行存储，构建分桶表

如果有两张表按照相同的划分规则，比如按照Join的关联字段，将各自的数据进行划分

在Join时，就可以实现Bucket与Bucket的Join，避免不必要的比较，减少笛卡尔积数量

创建普通表和分桶表，分的桶数最好是一样的，我们可以通过explain来进行查看

在使用之前需要开启分桶表

set hive.optimize.bucketmapjoin = true;

set hive.auto.convert.sortmerge.join = true;

set hive.optimize.bucketmapjoin.sortedmerge = true;
